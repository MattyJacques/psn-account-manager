# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A Chrome/Brave browser extension (Manifest V3) for managing multiple PSN account credentials. No build step, no dependencies, no package manager — plain HTML, CSS, and vanilla JavaScript loaded directly by the browser extension runtime.

## Development Workflow

**Load the extension:**
1. Open `chrome://extensions` or `brave://extensions`
2. Enable "Developer mode"
3. Click "Load unpacked" and select this directory
4. Click the refresh icon on the extension card after any code change

There are no build commands, tests, or linters configured.

## Architecture

The extension has two entry points:
- **Popup:** `popup.html` + `popup.css` + `popup.js` — all UI and account management
- **Background service worker:** `background.js` — handles the `openSignIn` message from the popup; creates a tab navigating to `https://www.playstation.com/en-gb/`, waits for it to fully load, then drives the toolbar (`driveToolbar()`): if an account is already signed in (profile icon matched by `[data-qa$="profile-icon#image#image"]`), it resolves `"needs-logout"` — it does **not** click Sony's sign-out button, because a server-side sign-out revokes the SSO session and invalidates that account's NPSSO token. Instead `driveSignInFlow()` responds to `"needs-logout"` by calling `clearSonyCookies()`, which deletes the specific Sony/PlayStation login + SSO cookies from the local cookie store (Sony's server is never told to log out, so the previous account's NPSSO stays valid), then **fresh-navigates** the tab to `https://www.playstation.com/en-gb/` (a `chrome.tabs.update`, not a soft reload — a reload can restore the signed-in page from the bfcache) and re-runs the toolbar driver; otherwise it clicks the sign-in button (matched by `[data-qa$="signin-button"]` with a `button.web-toolbar__signin-button` fallback, since Sony periodically renames the `data-qa` container path); `openSignIn` requires the toolbar drive to end in `"signin-clicked"` (up to two needs-logout→cookie-clear→re-drive cycles) and throws a descriptive error otherwise, rather than silently waiting out the 30s auth-tab timeout. If the cookie-clear + re-navigate fails to reach a logged-out toolbar within those two cycles, the account **fails** (the `outcome !== "signin-clicked"` throw) — it never falls back to a sign-out click, so no account's NPSSO is ever revoked. The cookies cleared (verified minimal, all removable via the three existing host permissions — no new host perms) are `npsso`/`JSESSIONID`/`dars`/`KP_uIDz-ssn` on `ca.account.sony.com`, `KP_uIDz-ssn` on `my.account.sony.com`, and `isSignedIn`/`pdccws_p`/`session`/`userinfo`/`pdcsi`/`pdcws2` on `.playstation.com` (removed via the `www.playstation.com` URL). It then waits for the Sony auth tab (`my.account.sony.com`) to open and injects a script that fills the email field (polling up to 20s — the tab's "complete" status fires well before the auth SPA renders the field, so a slow load needs a long window), clicks the email-submit button, waits up to 10s for the SPA to swap to the password step, fills the password field, and clicks the password-submit button. The injected script resolves with how far the fill got; anything short of `"submitted"` (e.g. `"email-input-timeout"`) makes `driveSignInFlow` throw immediately, so refresh-all retries the account right away instead of the untouched email page silently burning the full 90s capture window. After submission, `watchAuthInterstitials()` polls the auth tab for pages Sony can interpose: a passkey-creation prompt (auto-dismissed by clicking its `[data-qa="button-remind-later"]` button), the 2FA verification-code page (focuses the tab for manual entry, extends the capture timeout, and keeps watching so a passkey prompt shown after code entry is still dismissed), and the transient "Can't connect to the server." banner (`span[data-qa="message"]`, matched by text without the apostrophe) that sometimes fails the password submit — recovered by re-clicking the Sign In button, capped at 3 retries with a 3s cooldown between clicks; if the banner outlives the retries (two consecutive sightings after the last click), the watcher fails the capture immediately via `stopCapture(false)` rather than waiting out the 90s capture timeout, so refresh-all can start a fresh attempt. The watcher exits via an explicit `tabs.get` URL check once the auth tab lands back on playstation.com — `executeScript` does NOT throw on that navigation (the extension holds host permission there), so without the check the loop would poll the signed-in homepage for its whole remaining window (~30s, or ~5 min after a 2FA page stretched it), stalling the refresh-all queue after each sign-in

**Profile / NPSSO capture timing (subtle — do not move):** `openSignIn()` arms the capture (`startCapture()`, which registers the `interceptor.js` content scripts) **only after** the local logout (cookie-clear) and clicking sign-in — NOT before the initial page load. If it armed earlier, the interceptor would capture the *previously* signed-in account's `getProfileOracle` on the first load (storing the wrong `accountId`/`onlineId` and reading the wrong account's npsso), then `stopCapture()` after that first hit — before the new account ever signed in. Arming after sign-in means only the new account's post-auth homepage load is captured, and by then `account.sony.com`'s SSO session is the new account too.

**Capture fallback + identity guard:** the interceptor can miss the post-auth load (observed in practice — e.g. the return navigation restores the homepage from the back/forward cache, where content scripts don't re-inject and the fetch hook is absent). So after `watchAuthInterstitials()` returns, if the capture is still pending, `captureProfileFallback()` replays the profile query itself: it reads the full `getProfileOracle` request URL (persisted-query hash included) from the page's resource-timing timeline via `executeScript`, polls up to 10s for the SPA to issue it, and fetches it with `credentials: "include"` plus a `content-type: application/json` header (required by Apollo's CSRF guard; cookies carry the auth, so the response is always the *currently* signed-in account). Both the interceptor message and the fallback funnel into `handleProfileCaptured`, which is guarded against re-entry (`captureHandling`). Before saving, `persistCapturedProfile` rejects a capture whose `accountId` contradicts the row's previously captured `accountId` (a Sony SSO session that survives the local logout — e.g. if `clearSonyCookies()` misses a renamed cookie — can bounce the *previous* account straight back in, and saving would overwrite the row with another account's identity and token; the guard stays as defense-in-depth even though the cookie-clear normally prevents the bounce); the popup clears a row's captured identity fields when its email is edited so a legitimate account swap can't trip this guard. `refreshAll` also salvages an attempt whose flow threw (e.g. "Timed out waiting for Sony auth page" after an SSO bounce that never settled on the auth page) but whose capture landed anyway, by peeking at `captureDone` via a `Promise.race` against an already-settled `false` — it resets `captureDone` before each attempt so a flow that throws before `startCapture` re-arms can't read the previous account's result.

**Refresh All (sequential queue):** the popup header's `#refreshAllBtn` sends `refreshAll`; `refreshAll()` in the background loops over every stored account **one at a time** (the capture pipeline is a single slot — `pendingAccountId`), reusing `openSignIn()` per account. It knows an account is *finished* by awaiting `captureDone` — a promise created by `startCapture()` and resolved by `stopCapture(captured)` with `true` from `handleProfileCaptured` or `false` on capture timeout. A failed account (an `openSignIn()` throw or a `false` capture — e.g. the transient server banner) is **retried from scratch up to 3 attempts** (`ACCOUNT_ATTEMPT_LIMIT`), closing the dead attempt's tabs and waiting 5s between attempts (a throwing `openSignIn` attaches the tabs it created to the error as `err.tabIds` so the retry can close them too); the queue stops only when an account exhausts its attempts, leaving the last attempt's tabs open for inspection. A successful account's sign-in tabs (returned by `openSignIn()` as an id array — the auth step can spawn its own popup) are closed before the next account starts. Progress is published to `chrome.storage.local` under `"psn_refresh_state"` (`{running, index, total, activeId}`, removed when idle) because the queue cannot live in the popup — creating the sign-in tab closes the popup. The popup renders it as a `2/5` counter on the button next to a spinning SVG refresh icon (click again = cancel after the current account) and the FETCHING badge on the active row; the button label text lives in `#refreshAllLabel` so updates don't wipe the inline SVG. Single GET/SYNC is locked out while a run is active and vice versa. The worker clears `"psn_refresh_state"` at startup: a run only exists in one worker instance's memory, so a fresh worker means any persisted state is stale. While a capture is armed, a keepalive interval calls `chrome.runtime.getPlatformInfo` every 20s — Chrome kills an idle MV3 worker after ~30s and a bare pending `setTimeout` is not "activity", so without the tick a missed capture would let the worker (and the 90s capture timeout, and the whole queue awaiting `captureDone`) die silently mid-run.

**Check All (sequential queue):** the popup header's `#checkAllBtn` sends `checkAllNpsso`; `checkAllNpsso()` loops over every stored account that has an `npsso`, one at a time (the DNR rule is a single slot), validating each stored token via `checkNpsso()` and writing `npssoValid`/`npssoCheckedAt` as each result lands. An indeterminate result (network failure) records nothing and the loop continues — only cancellation (`cancelCheckAll`) stops it early. Progress is published to `chrome.storage.local` under `"psn_check_state"` (`{running, index, total, activeId}`, removed when idle and cleared at worker startup, same staleness reasoning as `"psn_refresh_state"`). A check-all run is mutually exclusive with single sign-in and refresh-all in both directions (`checking` flag ↔ `refreshing`/`pendingAccountId`), so a validity check never overlaps a sign-in's capture pipeline.

**Data flow:**
- All state lives in `chrome.storage.local` under the key `"psn_accounts"` as a JSON array
- `loadAccounts()` / `saveAccounts()` are the only storage access points
- `renderAccounts()` rebuilds the entire account list DOM from the in-memory array on every state change — there is no partial/incremental update
- `"psn_refresh_state"` and `"psn_check_state"` are the other two storage keys — refresh-all and check-all queue progress respectively, each written solely by the background worker and read-only in the popup

**Account object shape:**
```js
{ id, label, email, password, notes, createdAt, updatedAt?,
  accountId?, onlineId?, profileFetchedAt?, npsso?, npssoFetchedAt?, avatarUrl? }
```
- `id` comes from `uid()` (timestamp + random suffix)
- `password` is stored in plaintext in `chrome.storage.local`
- `accountId` / `onlineId` / `profileFetchedAt` are captured from the profile GraphQL response after a successful sign-in
- `npsso` / `npssoFetchedAt` hold the Sony SSO token fetched (also in plaintext) after that same sign-in — see below
- `avatarUrl` is the account's PSN avatar image URL, scraped by `fetchAvatarUrl()` from the toolbar profile icon (`img[data-qa$="profile-icon#image#image"]`, polled while the SPA renders it) on the signed-in playstation.com tab. It runs in `handleProfileCaptured` in parallel with `fetchNpsso()`, so it lands in the same storage write; a miss stores nothing and never breaks the capture

**DOM pattern:**
- `els` object caches all static DOM references at startup
- Dynamic account rows are `div.acct` elements built in `renderAccounts()` — no table
- Each row (click to expand, tracked by the in-memory `expandedId`) shows: a status left-border + badge derived from `npssoStatus()` (NOT FETCHED / NPSSO ACTIVE / EXPIRING SOON at 51 days / EXPIRED at 61 days — the ~61-day NPSSO lifetime — plus a transient FETCHING while `fetchingId` is set), an avatar (the captured PSN avatar image when `avatarUrl` is set, overlaying a gradient rounded square with initial that doubles as the fallback — the `<img>` removes itself on load error), label, PSN ID and masked-NPSSO lines, a time-ago stamp, and a GET/SYNC button. Expanding reveals the full account ID and NPSSO token plus COPY NPSSO / EDIT ACCOUNT / DELETE action buttons
- The GET/SYNC button sends `{action: "openSignIn", email, password}` to the background service worker, which opens `www.playstation.com/en-gb/` and programmatically drives the full sign-in flow (email entry → password entry → submit) — opening via the PlayStation homepage preserves all OAuth params that a direct URL to the Sony auth page cannot replicate. **The stored password is passed to the background worker and injected directly into the Sony auth page via `scripting.executeScript()`.** `fetchingId` is cleared when the background's storage write triggers the popup's `storage.onChanged` re-render
- The inline form (styled per the Claude Design mock) has Label / Email / Password fields — `onlineId` and `accountId` are auto-captured only (not on the form; `onlineId` shown read-only in the collapsed row, `accountId` in the expanded detail view) — and toggles between "add" and "edit" modes via `startEdit()` / `resetForm()`
- The form has its own close (`#cancelBtn`) and Cancel (`#cancelBtn2`) buttons; the `#addBtn` header button also closes the form if already open
- The popup loads Manrope / JetBrains Mono from Google Fonts (allowed by the default MV3 extension-page CSP; falls back to system fonts offline)

## Permissions

`manifest.json` declares:
- `storage`, `clipboardWrite` — core popup features
- `tabs`, `scripting` — required for the background service worker to create a tab and inject the sign-in automation
- host permission `https://www.playstation.com/*` — required for `scripting.executeScript()` to click the sign-in button on the PlayStation homepage
- host permission `https://my.account.sony.com/*` — required for `scripting.executeScript()` to fill credentials on the Sony auth page
- host permission `https://ca.account.sony.com/*` — required to read the account's NPSSO token from `https://ca.account.sony.com/api/v1/ssocookie` once the session is live. `fetchNpsso()` opens that endpoint as a background tab **in the sign-in window** and reads the `{ "npsso" }` JSON back, then closes the tab. It must NOT be a service-worker `fetch()` (the shared worker uses the default cookie store, so it returns a stale account's token when the user signs in inside a private window) nor an injected fetch from the playstation.com tab (cross-site to `account.sony.com`, so SameSite strips the Sony cookies). The tab runs in `handleProfileCaptured` (the confirmed-signed-in signal), so `npsso` is persisted in the same write as the profile fields.
- `declarativeNetRequestWithHostAccess` — required only for the NPSSO validity check (`checkNpsso()`). `fetch()` cannot set a `Cookie` header, so validating a *stored* token means injecting `Cookie: npsso=<token>` onto a single request to `https://ca.account.sony.com/api/v1/ssocookie` via a temporary DNR **session rule** (scoped to `resourceTypes: ["xmlhttprequest"]` so it can't touch `fetchNpsso()`'s tab navigation), then removing it. The rule reuses the already-declared `ca.account.sony.com` host permission and never touches the browser's cookie store, so a check can't disrupt a live session. Do not switch this to the broad `declarativeNetRequest` — keep the surface minimal. (The `cookies` permission is declared separately, for `clearSonyCookies()` — the DNR validity check itself still never reads or writes the cookie store.)
- `cookies` — required by `clearSonyCookies()` to switch accounts without invalidating NPSSO tokens. Account switching deletes the Sony/PlayStation login + SSO cookies from the **local** cookie store instead of clicking Sony's sign-out button (a server-side sign-out revokes the SSO session, killing the previous account's NPSSO). Every cookie is removed via a URL covered by an **already-declared** host permission (`www.playstation.com`, `my.account.sony.com`, `ca.account.sony.com`), so this adds **no new host permissions**. It only ever *deletes* named cookies — it never reads the cookie store or touches the server session, so the token stays valid server-side. Empirically verified: a local clear leaves the NPSSO ACTIVE.

Do not add `clipboardRead`, `activeTab`, or broader host permissions — keep the surface minimal.
